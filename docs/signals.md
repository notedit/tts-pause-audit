# 5 类停顿信号详解

停顿检测器输出的 5 类信号，每类对应一种听感问题，组合起来覆盖所有"哪里停顿了 / 停得对不对"的情形。

## 一张速查表

| 信号 | "我抓的是" | 默认严重度 |
|---|---|---|
| `S1 inter_char` | 不该断的地方断了 | ★★★（最常见的 TTS 毛病） |
| `S2 char_too_long` | 字被拖长，藏了静音 | ★★（反向定位的关键） |
| `S3 in_word` | 一个字被读破 | ★★★（极少见但绝对刺耳） |
| `S4 long_pause` | 标点处停太久 | ★（多数是自然的，需 LLM 判） |
| `S5 leading/trailing` | 头尾的静音 | —（仅记录，非异常） |

`suspicious=True` 的 finding（S1/S2/S3/S4）会被自动送 LLM 复核，得到最终的 ✓ 自然 / ❌ 不自然 判定。S5 不送 LLM。

## 全景图

原文："**刚收到快递，发现是男朋友偷偷寄来的生日蛋糕。**"

```
时间 →
[___] 刚·收·到·快·递 [|] 发·现·是·男·朋·友 [▮▮▮] 偷·偷·寄·来·的 [▮▮▮] 生·日·蛋·糕 [|] ...
 S5         字间正常           S4         字间正常              S1            字间正常               S4
```

- `·` 正常发声的字
- `[|]` 标点位置（合理停顿 → S4）
- `[▮]` 句中实际静音（看谷在哪里 → S1 / S2 / S3）
- `[___]` 整段音频首尾安静段（S5）

5 类信号本质上回答 5 个不同的问题：

| 信号 | 一句话 | 它在问什么 |
|---|---|---|
| S1 inter_char | 字与字之间出现真实静音段 | "这里**该不该**有停顿？" |
| S2 char_too_long | 一个字本身的时长被拉得异常长 | "这个字怎么读了这么久？" |
| S3 in_word | 静音段**落在某个字内部** | "这个字是不是被读断了？" |
| S4 long_pause | 标点位置上的停顿过长 | "这个逗号/句号停得太久了吗？" |
| S5 leading/trailing | 整段音频开头或结尾的安静 | "前后多余的安静要不要剪掉？" |

---

## S1 inter_char — "字之间出现了停顿"

**检测条件**：两个相邻字之间，RMS 能量谷 ≥ 260ms 且比说话能量低 ≥ 12dB（**或** 前一个字时长 z ≥ 2.0）。

**例子**：

> 男朋**友** ▮▮▮ **偷**偷寄来的（友→偷 350ms）

原文里 "男朋友" 和 "偷偷" 之间没有任何标点，但实际却有 350ms 的真实静音 → 不该出现的停顿 → 触发 S1。

**为什么这是核心信号**：TTS 输出最常见的毛病就是"在不该断的地方断"。S1 直接捕捉这个。

## S2 char_too_long — "某个字读得异常久"

**检测条件**：某字时长 - 平均字时长，除以标准差，**z ≥ 2.0**。

**为什么需要这个信号**：很多 ASR 在做强制对齐时会把"字尾的静音"吞进那个字的时长里。比如一个 160ms 的"修"字被对齐成 420ms——这中间 260ms 其实是静音，但能量谷在字框内部没法被 S1 看到。

**例子**：

> "不给保**修** [拖到 420ms]。这明摆着..."

字 "修" 的时长被算成 420ms，z = +2.2。我们检测到 z 异常之后，再去看那段时间内有没有 RMS 能量谷，用谷的真实起止做"反向定位"，得到一个 390ms 的实际静音段。

## S3 in_word — "字内部出现静音 = 破词"

**检测条件**：RMS 能量谷**完全落在一个字的内部**（`valley.start > char.start + 60ms` 且 `valley.end < char.end - 60ms`），谷 ≥ 120ms 且深度 ≥ 18dB。

**为什么需要这个信号**：TTS 偶尔会把一个字读到一半"卡"一下，比如 "保**(停顿)**修"——一个字被切开读。这种情况能量谷既不在字间也不在标点位，只能用"谷在字内"来判定。

**例子**（典型场景）：

> "保 [▮▮] 修" — "保" 0.20-0.55s，能量谷 0.30-0.42s（120ms） → 字 "保" 被读破

判定结果：**破词，永远不自然**。

## S4 long_pause — "标点位置的停顿，但停太久"

**检测条件**：两个相邻字之间，原文有标点，**且** gap ≥ 260ms。

**这是与 S1 的对照信号**。原文里有标点 → 这里**本来就该停顿**；但如果停得太久（比如逗号停了 600ms），听感上也可能怪，需要 LLM 判定。

**例子（自然）**：

> "蛋**糕**。 ▮▮▮ **虽**然隔着..."（糕→虽 320ms）

"。" 在这个位置出现 → S4 触发；LLM 判定 → ✓ 自然（完整子句之间停 320ms 合理）。

**例子（不自然）**——0021：

> "违反规**则**， ▮▮▮▮ **在**先还..."（则→在 480ms）

原文有 "，"，也是 S4，但 LLM 判定 → ❌ 不自然（"违反规则在先"是一个连贯短语，被这个 480ms 切碎了）。

## S5 leading / trailing — "整段开头/结尾的安静"

**检测条件**：
- `leading`：第一个字的 `start_time > 0`（开头有静音）
- `trailing`：最后一个字的 `end_time < total_duration`（结尾有静音）

**例子**：

> [⏮ 320ms 静音] 刚收到快递... [...嘴。⏭ 490ms 静音]

这两段静音**不算异常**（很多 TTS 都会在前后留 buffer），只是记录下来，方便：
- 自动剪辑去掉头尾
- 判断录制工具是不是有错位

它们的 `suspicious=False`，不会被送 LLM 判定。

---

## 5 类信号怎么协作 + 怎么去重

它们之间不是孤立的，举个例子：

**0021 错→这**：
1. 字 "错" 时长 320ms（z=2.5）→ **S2 char_too_long** 想报警
2. 同时 "错" 后面有 "。"，gap 640ms → **S4 long_pause** 也报警

如果两个都报会导致同一位置发两次 LLM 调用。所以 detect 阶段的 dedup 规则是：

- **S4 先发**（标点位置的停顿，信号更"硬"）
- **S2 再看**：如果发现自己的位置已经被 S4 覆盖，跳过

最终 0021 错→这 只会有 **1 条 S4 long_pause**，被 LLM 判定为 ✓ 自然（完整子句间）。

## 一张映射图

```
       原文位置有标点？
            /          \
          是            否
          ↓             ↓
        gap ≥ 260ms?   gap ≥ 260ms 或 z ≥ 2.0？
        /     \           /            \
       是      否         是             否
       ↓       ↓          ↓              ↓
       S4    punct_*    S1 / S2        正常发音
                      （取决于谷在哪）

字内部出现谷？        → S3 in_word
头/尾有静音段？       → S5 leading / trailing
```

## 实际阈值常量（`pause_detector/tools/detect_pauses.py`）

```python
CHAR_DUR_Z              = 2.0    # S2: 字时长 z-score 阈值
VALLEY_DROP_DB          = 10     # 能量谷判定：比 ref_db(p90) 低多少 dB
VALLEY_MIN_MS           = 80     # 谷的最小时长
INTER_CHAR_MIN_MS       = 260    # S1: 字间停顿最小时长
INTER_CHAR_MIN_DROP_DB  = 12     # S1: 字间停顿最小深度
IN_WORD_MIN_MS          = 120    # S3: 破词谷最小时长
IN_WORD_MIN_DROP_DB     = 18     # S3: 破词谷最小深度
PUNCT_PAUSE_MIN_MS      = 260    # S4: 标点位停顿最小时长
```

这些阈值在 `pause_detector/tools/detect_pauses.py` 顶部集中定义，移植自原 DashScope 参考实现 `docs/legacy/pause_detect_dashscope.py` 的校准值。
